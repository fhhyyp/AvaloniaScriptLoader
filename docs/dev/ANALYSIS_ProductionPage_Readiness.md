# 生产级页面就绪度分析

## 分析方法

以一个典型的生产级页面为基准 —— **用户管理系统**（含列表、搜索、表单、校验），逐项评估当前框架是否足以实现。

### 目标页面功能

```
┌─────────────────────────────────────────────┐
│  用户管理                          [+ 新增] │
├─────────────────────────────────────────────┤
│  🔍 [____搜索用户名____]  [筛选▾] [刷新]    │
├─────────────────────────────────────────────┤
│  ┌──────┬──────────┬────────┬─────────────┐ │
│  │ 头像 │ 用户名    │ 角色   │ 操作        │ │
│  ├──────┼──────────┼────────┼─────────────┤ │
│  │ 🧑   │ 张三     │ 管理员 │ [编辑][删除] │ │
│  │ 👩   │ 李四     │ 编辑   │ [编辑][删除] │ │
│  │ 🧑   │ 王五     │ 用户   │ [编辑][删除] │ │
│  └──────┴──────────┴────────┴─────────────┘ │
│                          ← 1  2  3 ... 9 →  │
├─────────────────────────────────────────────┤
│  点击"新增"弹出对话框：                      │
│  ┌─────────────────────────┐                │
│  │ 用户名: [____________]  │ ← 校验: 不为空  │
│  │ 邮箱:   [____________]  │ ← 校验: email格式│
│  │ 角色:   [管理员▾]       │                │
│  │         [保存] [取消]    │                │
│  └─────────────────────────┘                │
└─────────────────────────────────────────────┘
```

### 逐项能力评估

---

## 1. 布局能力

| 需求 | 当前支持 | 差距 |
|------|----------|------|
| 顶部工具栏（水平布局） | ✅ StackPanel horizontal | — |
| 搜索框 + 按钮行 | ✅ StackPanel horizontal | — |
| 主内容区（表格/列表） | ⚠️ ListBox (仅字符串) | 🔴 不支持多列数据行 |
| 底部分页栏 | ✅ StackPanel horizontal | — |
| 弹窗对话框 | ✅ app.showMessage (单向) | 🟡 无自定义内容对话框 |
| 表单布局（标签+输入框） | ✅ Grid + FormField 组件 | — |
| 响应式布局（窗口缩放自适应） | ⚠️ Grid Star 可用 | 🟡 无 Grid 自适应断点 |

**结论：基础布局可用，但缺少 DataGrid/多列列表。**

## 2. 控件覆盖

| 需求 | 当前支持 | 替代方案 |
|------|----------|----------|
| 文本输入 | ✅ TextBox | — |
| 下拉选择 | ✅ ComboBox | — |
| 按钮 | ✅ Button | — |
| 复选框 | ✅ CheckBox | — |
| 图片/头像 | ✅ Image | — |
| 标签/文本 | ✅ Label | — |
| 多列数据表 | ❌ DataGrid | 🟡 需用 Grid + vfor 手动拼 |
| 日期选择 | ❌ DatePicker | 🔴 需新增 |
| 数字输入 | ⚠️ TextBox | 🟡 无数字专用控件(NumericUpDown) |

**结论：7 个控件缺失中，DataGrid/DatePicker/ProgressBar/Menu 对生产页面最关键。**

---

## 3. 数据绑定

| 需求 | 当前支持 | 差距 |
|------|----------|------|
| 表单字段 ↔ 数据双向同步 | ✅ inpc twoway | — |
| 列表 ↔ 数据源同步 | ⚠️ vfor + inpc array | 🟡 仅支持整体替换 |
| 搜索过滤 | ⚠️ 手动 computed | 🟡 无内置 filter 管道 |
| 分页状态管理 | ⚠️ 手动 inpc | 🟡 无内置分页组件 |
| 远程数据加载 | ❌ 无 fetch API | 🔴 数据只能硬编码 |
| 加载状态 | ❌ 无 loading 状态管理 | 🔴 无法显示 spinner/骨架屏 |
| 空状态 | ⚠️ vif 可实现 | — |
| 错误状态 | ❌ 无 try/catch 在脚本中 | 🔴 网络错误无法捕获展示 |

**结论：最大的阻塞项 —— 无法从远程加载数据，所有数据必须硬编码在脚本中。**

---

## 4. 表单校验

| 需求 | 当前支持 | 差距 |
|------|----------|------|
| 必填校验 | ⚠️ 手动 computed + vif | 🟡 无内置校验框架 |
| 格式校验(email/phone) | ❌ 无正则/格式函数 | 🔴 需手动实现字符串检查 |
| 校验错误展示 | ⚠️ vif + label 变红 | 🟡 无内置错误样式类 |
| 提交前整体校验 | ⚠️ 手动 if/else | 🟡 无 validate() 聚合函数 |
| 实时校验(输入时) | ✅ onChange + inpc | — |
| 失焦校验 | ❌ 无 onBlur 事件 | 🟡 需新增事件 |

**结论：可手动实现但代码量巨大，缺少内置校验原语。**

## 9. 性能

| 需求 | 当前支持 | 差距 |
|------|----------|------|
| 大数据列表(1000+行) | ⚠️ vfor 全量渲染 | 🔴 无虚拟化，1000 行会卡顿 |
| 列表增量更新 | ✅ vfor index-diff | — |
| 搜索防抖 | ❌ 无 setTimeout | 🔴 每次按键都触发重渲染 |

**结论：无虚拟化列表 + 无防抖 = 中大型列表会卡顿。但 IDisposable 已就位，内存可控。**

## 差距汇总

| 等级 | 数量 | 阻塞项 |
|------|------|--------|
| 🔴 致命 | **6** | fetch() HTTP、键盘事件、自定义对话框、DataGrid、列表虚拟化、样式复用 |
| 🟡 重要 | **14** | 防抖、onBlur、校验框架、空/加载/错误状态、localStorage、路由、ToolTip、DatePicker、ProgressBar、Menu、数字输入、响应式布局、分页、定时器 |

## 判断

**当前框架处于「Demo 级」→「生产级」的临界点。**

- ✅ 可以构建：简单 CRUD 页面（静态数据、无校验、无网络）
- 🔴 无法构建：真实业务页面（数据来自 API、有表单校验、键盘交互、大数据列表）

### 最小可行提升路径（6 项致命修复）

| # | 能力 | 实现方式 | 复杂度 |
|---|------|----------|--------|
| 1 | **fetch() HTTP** | 注入 `fetch(url, opts)` → `HttpClient` 封装，返回 `ObjectValue {status, body, json()}` | 🟢 低 |
| 2 | **键盘事件** | 新增 `onKeyDown` 事件 → `Control.KeyDown`，事件参数含 `e.key` | 🟢 低 |
| 3 | **自定义对话框** | `app.showDialog(contentDescriptor)` → 模态 Window + 返回值 | 🟡 中 |
| 4 | **DataGrid** | 新增 DataGrid 控件 + 列定义 API | 🟡 中 |
| 5 | **列表虚拟化** | vfor 内使用 `VirtualizingStackPanel` 替代 StackPanel | 🟢 低 |
| 6 | **样式复用** | `style(name, props)` 注册 + `"class" = "name"` 属性 | 🟢 低 |

这 6 项全部实现后，即可支持用户管理页面（含 API 数据加载、搜索、表单校验、分页列表）的生产级开发。
